Introduction

The yocto-autobuilder is a continuous integration system as well as the build and release tool used by the Yocto Project and its end users. The main purpose of the yocto-autobuilder is to replicate many of the steps that an end user would take to build an embedded Linux distribution, test the build output of a build run and perform most of the Yocto Project release.  The current production yocto-autobuilder is found at http://autobuilder.yoctoproject.org/main/

The core of the yocto-autobuilder is Buildbot, currently at 0.8.8 with a few Yocto Project specific patches. However the yocto-autobuilder is not like most Buildbot implementations. Most Buildbot implementations treat Buildbot as an out of the box CI solution.

The yocto-autobuilder utilizes Buildbot as set of libraries. Think of the yocto-autobuilder as an abstraction layer around Buildbot that allows you to interact with Buildbot and generate Buildbot configuration files without actually having to know the intricacies around Buildbot. For more information on design philosophies regarding the yocto-autobuilder see the section “High Level Architectural Overview: Introduction and Philosophy”

The yocto-autobuilder was initially written by Richard Purdie <richard.purdie@linuxfoundation.org>. In December 2012, it was entirely rewritten by Elizabeth ‘pidge’ Flanagan <elizabeth.flanagan@intel.com>, who is the current maintainer.

The current production Yocto Project yocto-autobuilder infrastructure is maintained by both Elizabeth Flanagan and Michael Halstead <michael@yoctoproject.org>. The majority of discussion related to the yocto-autobuilder occurs on the Yocto Project mailing list <yocto@yoctoproject.org> as well as the #yocto channel on FreeNode irc servers.

The yocto-autobuilder code is licensed under the GNU General Public License version 2 (GPL-2.0). Dependencies packaged with the yocto-autobuilder are licensed under their associated licenses as denoted in the COPYING* files within the yocto-autobuilder root directory.

Contributing to the yocto-autobuilder

The yocto-autobuilder is an open source project and as such we welcome community contributions. Because the yocto-autobuilder is extremely configurable and flexible, we recognize that developers will want to extend, configure or optimize it for their specific uses.

We ask that if you contribute a patch to the yocto-autobuilder that you make the patch as generic as possible. If you have, for example, written a buildstep that is useful to your organization and is probably useful to the overall project, we ask that you make it so that specific things (strings, emails, etc.) that would be specific to your organization are abstracted out into the global configuration file (autobuilder.conf.example).

Getting Started

The quickest method to get a basic yocto-autobuilder instance in place is to do the following:

git clone git://git.yoctoproject.org/yocto-autobuilder
cd yocto-autobuilder
. ./yocto-autobuilder-setup
yocto-start-autobuilder both

 

This will set up a basic yocto-autobuilder controller and a single worker on a single machine. The yocto-autobuilder utilizes (generally) a single controller and one or more workers. These systems can be started and stopped independently of each other. In order to start or stop either the worker or controller independently:


. ./yocto-autobuilder-setup
yocto-stop-autobuilder worker
yocto-stop-autobuilder controller

. ./yocto-autobuilder-setup
yocto-start-autobuilder worker
yocto-start-autobuilder controller

Please be aware that you MUST either setup a ~/.bashrc with the contents of ./yocto-autobuilder-setup or make it so that the file is sourced prior to trying to starting or stopping any portion of the yocto-autobuilder.

Do keep in mind that this is a very basic setup, with a single machine, single controller and a single worker utilizing the default buildsets we use for Yocto Project releases. These buildsets are rather large and are meant to be utilized in a large scale cluster.

Most instances of the yocto-autobuilder are generally part of a large scale infrastructure utilizing many workers. For details on how to set up and maintain large scale yocto-autobuilder infrastructures like this, please see the section “Advanced Topics: Maintaining and Setting up yocto-autobuilder clusters.”

During the initial run of . ./yocto-autobuilder-setup a generated username and password will be created. This is to ensure that some level of security is enabled during the initial configuration.

Setting up automated testing

The yocto-autobuilder has the ability to run the Yocto Project automated test suites. This is run by the RunSanityTests build steps.

In order to be able to run these, you must set up your workers to be able to run qemu with virtual network tap devices. There are a few ways of setting these up, but probably the best method to do so securely is to utilise the runqemu-gen-tapdevs script in scripts directory of the the poky repo.

Setup prerequisites:
     -    you will need a checkout of poky (git clone git://git.yoctoproject.org/poky)
     -    you will need a native-sysroot (we suggest building your own or downloading one from the yocto project download site at downloads.yoctoproject.org/releases/yocto)

  • you will need to disable NetworkManager as it seems to remove tap devs

You will need to run the script as root:

 sudo ./scripts/runqemu-gen-tapdevs <uid of the build user> <gid of the build user> <num of tap devices. we use 24 on our servers> <native-sysroot-basedir>

Once this is complete, you should setup a vnc session and log into it at least once.

1. Install tight-vnc client and server.

2. Set up tap devs by running poky/scripts/runqemu-gen-tapdevs

3. Add "xterm*vt100*geometry: 80x50+10+10" to .Xdefaults

4. Setup and start vnc session as the autobuilder user.

5. You MUST manually connect to the vnc session at least once prior to

   running a qemu sanity test (Something is getting set during the initial

   connection that I haven't figured out yet. Manually connecting seems to

   set up the session correctly.)


Introductory use of the yocto-autobuilder

Following the above steps you should have a functional yocto-autobuilder running on your host system. You should be able to point a web browser to port 8010 of the machine you have set the yocto-autobuilder up on and see a functional autobuilder (i.e. http://localhost:8010 or http://myawesomeautobuilder.foo.org:8010)

High Level Architectural Overview

Introduction and Philosophy

The yocto-autobuilder was initially a standard Buildbot implementation. In it’s initial incarnation it was a simple 600 line configuration file. As the needs of the Yocto Project grew, the yocto-autobuilder was relied on more and more to support the project. It became apparent in mid-2012 that the initial design of the yocto-autobuilder wasn’t flexible enough to deal with the needs of the project.

The re-write of the yocto-autobuilder started out with a few design goals:

  1. The yocto-autobuilder is not a ‘reference design’. It is a product and should be developed as if it were used by those outside the Yocto Project.
  2. Casual users of the yocto-autobuilder should never have to see the inner workings of Buildbot.
  3. Developer users of the yocto-autobuilder should have a stable and standard API to utilize that limits the amount of Buildbot knowledge they need to have.
  4. Code, configuration and buildset definitions will never again be in a single unmaintainable file.
  5. Conditional buildsteps will always be written at the buildstep layer. We will never utilize doStepIf.
  6. Buildsteps should have a clearly defined standard.
  7. Buildset definitions should be written as configuration, not code.
  8. We will support the ability to build prior Yocto Project releases for a minimum of what was initially a year (we can currently build with minor issues as far back as 1.4)
  9. The yocto-autobuilder cluster used by the Yocto Project will share between worker nodes as much common data (SSTATE_DIR, DL_DIR) as possible in order to speed up build times.


The current yocto-autobuilder code base has been operational since the 1.3 release. The Yocto Project autobuilder infrastructure consists of a NAS which contains the SSTATE_DIR and DL_DIR common to all builders operating on all workers, one controller and ten physical machines running two to three workers a piece. This allows us to build, test and release, around .25 TB of build artifacts and around 250,000 build artifacts (around 140 images across 17 different MACHINES) in around 3 to 11 hours depending on the size of the changeset being built and the number of things that need to be rebuilt due to the change.

How do I change the global configuration file?

Change the values of a field by editing the value to the right of the ‘=’ sign. Also, some fields may be commented out via a ‘#’ sign. These can be removed to make the property take effect. Make sure to restart the autobuilder after changing anything in the configuration file.

  • yocto-controller/controller.cfg file
  • yocto-worker/buildbot.tac file

Buildsets

What is a buildset?

A buildset is a configuration file that tells the yocto-autobuilder parser basic information and the buildsteps necessary to create a Buildbot build factory. The buildset directory’s contents are parsed during start of the yocto-autobuilder controller and a Buildbot configuration is generated from these files.

Buildset files are essentially python abstract syntax trees.


[buildset-name]
builders: ['example-worker', 'other-example-worker']
repos: [{'poky':
           {'repourl':'git://git.yoctoproject.org/poky',
            'branch':'daisy'}}]
props: [{'choice-box':
                 {'prop_type':'ChoiceStringParameter',
                 'name': 'choice-box',
                  'choice':['option-one', 'option-two'],
                 'label':'<hr><h3>CHOICE_BOX_HEADING:</h3>'}}]
steps: [{'SetDest':{}},
       {'CheckOutLayers': {}},
       {'RunPreamble': {}},
       {'CreateAutoConf': {'machine': 'example-machine', 'distro': 'poky'}},
       {'CreateBBLayersConf': {}},
       {'BuildImages': {'images': 'custom-example-image'}}]
scheduler: [{'name-of-scheduler': {'type':'Nightly',
                                       'repository':'poky',
                                       'hour':0,
                                  'minute':0}}]


Buildsets have at least three, possibly five sections within them - builders, repos, props, steps and schedulers with the props and schedulers sections being optional. Buildsets within the buildset-config directory will appear on the waterfall page of the autobuilder.

What is the format of a buildset?

builders

A list of workers that a buildset should use. If only one worker is being used by the buildset, a string can be used instead of a list.

builders: 'example-worker'

or

builders: ['example-worker', 'other-example-worker']

repos

A list of dictionaries that provides information about the repositories that are needed for a buildset. NOTE: Either the poky or oe-core repository must be listed first in the repos section because anything above them will be deleted when poky or oe-core is re-checked out, as they are the root directories for building.

       {'name-of-repo':
           {'repourl':'git://path.to/git_repo',
            'layerversion':'core',
            'checkout':False,
            'branch':'master'}},

Extra Notes on SingleBranchScheduler setup

There is one SBS scheduler for each repository in the buildset, i.e., if your buildset contains more repositories, and you want build triggered by changes to any of them, you need to define a scheduler for each. This also means that you can mix and match, e.g., only do nightly rebuilds for some repos and immediate for others.

The scheduler change filter is set up to watch changes only in the single repository specified, so you have to set up your repository hook to sent the repository url in the change set with the --repository argument to git_buildbot.py, e.g., your git post-receive  hook could look something like:


   #!/bin/sh
   echo "$1 $2 $3" | git_buildbot.py --repository="git://some_repo..."


For local testing, you can use a post-merge hook instead, along these lines:


   #!/bin/sh
   PRE=$(git rev-parse 'HEAD@{1}')
   POST=$(git rev-parse HEAD)
   SYMNAME=$(git rev-parse --symbolic-full-name HEAD)
   echo "$PRE $POST $SYMNAME" | \
   git_buildbot.py --repository="file:///where_your_repo_is"


Because of the way buildbot change sources work, your change user names must not collide with the user names used by your workers. .

If you are using a GitPoller changesource and you specify a valid repository but an non-existent branch, Buildbot will silently ignore your scheduler. If a repository has changed recently but your build has not triggered, validate that your repository and branch settings are valid.

For either a PBChangeSource or a GitPoller changesource, the scheduler will not see the initial repository discovery as a change. If you are using a SingleBranchScheduler schedule, it is good practice to force a build to verify the buildset works correctly.

Example of PBChangeSource:

[nightly-x86]

How do I create my own buildset?

  1. Make a new <buildset-name>.conf file and put it in the buildset-config directory.
  2. Add the [<buildset-name>] field with a name unique within the buildset-config directory. A best practice is to make the name of the file without the .conf and the name of the buildset the same. Example: foo.conf and [foo]
  3. Add the workers you want to build with to the builders field.
  4. Identify the repositories and branches needed to build what you want, and add them to the repos field. Either the poky or oe-core repository must be listed first in the repos section because anything above them will be deleted when poky or oe-core is re-checked out, as they are the root directories for building.
  5. If it needs any custom properties, add them to the props section.
  6. Identify the steps that are needed to complete your build. For basic builds, usually at least 'SetDest', 'CheckOutLayers', 'RunPreamble', 'CreateAutoConf', 'CreateBBLayersConf', and 'BuildImages' are used. Fill out these and any other relevant steps with the necessary information.

Buildsteps

A buildstep is a python class used to make BuildBot do a certain task.

There are a number of buildsteps available by default that will take care of most basic building needs. For other tasks you can create your own buildstep.

Creating your own buildstep is easy as long as you know about a few key functions, variables, and functions to override. First, import ShellCommand from buildbot.steps.shell and use that as your new buildstep’s type. For example, class Foo(ShellCommand). Then, use the table below.

One of the less common issues people run across is a build worker not connecting to the controller. There could be a lot of reasons for this, most of them involving an inability of the worker to connect to the controller port.

First, check the yocto-worker/twistd.log as well as the yocto-controller/twistd.log. These will generally give you a good idea as to what is going on. Next, check the yocto-worker/buildbot.tac file and ensure that the worker is pointed to the correct controller, port and utilizes the correct password. Lastly, check that the buildsets are referencing the name of the worker as defined in yocto-worker/buildbot.tac.

Maintaining and Setting up yocto-autobuilder clusters

The Yocto autobuilder is most useful when deployed in a large organisation. This allows not just a CI solution, but also allows for the autobuilder to provide a source mirror, an sstate mirror for developers to utilize, automated testing, buildhistory and error reporting integration.

In the above image, we have set up a cluster of workers. These workers all point to the same controller. They also share an NFS mount. (NOTE: It is imperative that all workers and the NAS utilize the same UUID as the build user!)

The workers will utilise the NAS as a shared DL_DIR/SSTATE_DIR. This enables a bit of speedup. It also ensures that we only keep one version of DL_DIR and SSTATE_DIR, thus minimizing disk space.

- From an IP protection standpoint, if you are building protected IP source, you do not want to expose your DL_DIR and SSTATE_DIR outside of your organisation.

Using EAM/AD with Yocto-Autobuilder

In the current version of yocto-autobuilder (Buildbot 0.8.8), using EAM/AD for authentication must be done via reverse proxy and Kerberos. Later versions of Buildbot are able to use LDAP for authentication but this is not implemented in 0.8.8. Hence, for this example configuration, reverse proxy and Kerberos are used here.

This example configuration is based on a Fedora 20 installation running Apache. This is also a “workstation” configuration. That is, it is assumed that one is trying to connect to a corporate or similar EAM/AD setup, and that the Kerberos server is running elsewhere.  Therefore, this documentation does not cover setting up a Kerberos server.

Set Up the Reverse Proxy for Apache

Setting up the reverse proxy is simple, and details are available here:

http://httpd.apache.org/docs/2.0/mod/mod_proxy.html - forwardreverse

 

You do not need to enable the forward proxy just to use reverse proxy. If you don’t need to use the forward proxy, it is best to disable it for the sake of security.  

To enable the reverse proxy only, add a section like this to your httpd.conf file.

ProxyRequests Off

<Proxy *>

   Order deny,allow

   Allow from all

</Proxy>

ProxyPass /buildbot/ http://myhost.somewhere.com:8010/

ProxyPassReverse /buildbot/ http://myhost.somewhere.com:8010/

Be mindful of the trailing slashes. If your autobuilder webStatus page doesn’t load, you have likely mismatched your slashes. Note that the port here is the autobuilder default. Adjust yours accordingly. This also assumes that Apache is running on the default port of 80.

Once you have that in place stop and restart Apache to load the new config.

 

 

Edit the Autobuilder Config

 

Now you need to tell the autobuilder about this change by editing yocto-autobuilder/yocto-controller/controller.cfg.  In the section labeled “PROJECT IDENTITY,” you need to set the URLs to correspond to your reverse proxy settings.

 

Example:

 

####################

# PROJECT IDENTITY #

####################

 

<snip>

 

c['title'] = "The Yocto Autobuilder"

c['titleURL'] = "http://myhost.somewhere.com:8010/buildbot/"

 

# the 'buildbotURL' string should point to the location where the buildbot's

# internal web server (usually the html.WebStatus page) is visible. This

# typically uses the port number set in the Waterfall 'status' entry, but

# with an externally-visible host name which the buildbot cannot figure out

# without some help.

 

c['buildbotURL'] = "http://myhost.somewhere.com:8010/buildbot/"

 

Note that this matches the URL in the reverse proxy config. If you drop a trailing slash here, your autobuilder page will not likely come up, or subpages will not come up properly.

Also note that the default for c['titleURL'] uses https. If you are not using https as your buildbotURL, you’ll want to change that.

Once this config is complete, you should now be able to load your autobuilder webStatus page, and subpages (waterfall, grid, buiders, etc.) using the newly defined URL: http://myhost.somewhere.com/buildbot

 

 

Configure Kerberos

 

Kerberos documentation may be found at http://web.mit.edu/kerberos/krb5-latest/doc/.

 

Fedora’s Security guide at https://docs.fedoraproject.org/en-US/Fedora/19/html/Security_Guide/sect-Security_Guide-Kerberos-Configuring_a_Kerberos_5_Client.html

A CentOS guide: https://www.centos.org/docs/5/html/5.1/Deployment_Guide/s1-kerberos-server.html

Detailed explanations of commands, etc., are available in the above references. Please refer to the documentation for a more info on the various steps/commands. Obviously, the first step is to install the required packages: krb5-libs, krb5-server, and krb5-workstation and any missing dependencies.

You will also need a working krb5.conf configuration or have the information know to create one.

Example krb5.conf file:

 

[logging]

  default = FILE:/var/log/krb5libs.log

  kdc = FILE:/var/log/krb5kdc.log

  admin_server = FILE:/var/log/kadmind.log

 

[libdefaults]

  default_realm = CORP.FOO.COM

  dns_lookup_realm = false

  dns_lookup_kdc = false

  ticket_lifetime = 24h

  renew_lifetime = 7d

  forwardable = yes

 

[realms]

 CORP.FOO.COM = {

  kdc = CORP.FOO.COM:88

  admin_server = corp.foo.com:<port>

  default_domain = corp.foo.com

 }

 

[domain_realm]

.example.com = CORP.FOO.COM

example.com = CORP.FOO.COM

 

 

Next, create the database using the kdb5_util command:

 

$ kdb5_util create –s

 

 

Then edit /var/Kerberos/krb5kdc/kadm5.acl. You should be able to use something along the lines of:

*/admin@EXAMPLE.COM  

 

Now you can create the first principal by running:

$ kadmin.local –q “addprinc <username>/admin”

 

 

Once you have completed that, start the services:

 

$ service krb5kdc start

$ service kadmin start

Now run kinit to test:

$ kinit <username>

 

You should be prompted for the user password, which should authenticate successfully using the user’s EAM/AD password.

You should now also have an /etc/krb.keytab file. You will need this for the Apache and the autobuilder to authenticate against.

 

 

Revisit Apache Conf

 

Apache needs to be configured to use Kerberos for authentication. You will need to add something similar to this to your httpd.conf file:

 

<Location "/">          

  AuthType Kerberos

  AuthName "Buildbot"

  KrbMethodNegotiate On

  KrbMethodK5Passwd On

  KrbAuthRealms CORP.FOO.COM

  KrbVerifyKDC off

  KrbServiceName Any

  Krb5KeyTab /etc/krb.keytab

  KrbSaveCredentials Off

  require valid-user

  Order allow,deny

 

  Satisfy Any

 

  #] SSO

  RewriteEngine On

  RewriteCond %{LA-U:REMOTE_USER} (.+)$

  RewriteRule . - [E=RU:%1,NS]

  RequestHeader set REMOTE_USER %{RU}e

 

</Location>

 

Adjust the path to your krb.keytab file and Realm accordingly.

 

Note: This config is copied straight from http://docs.buildbot.net/latest/manual/cfg-www.html with minor adjustments to genericize the Realm and path to the keytab file. Be aware that this doc is for a newer version of BuildBot and the backend auth component has been significantly rewritten. While this Apache conf works, the other configuration information (for plugins, etc) in this doc is NOT implemented in 0.8.8.

 

Restart httpd services to pick up the new config.

Revisit Autobuilder Config

 

Lastly, revisit the autobuilder config to set the path to the krb.keytab file. In the “STATUS TARGETS” section of yocto-controller/controller.cfg, adjust the path accordingly. Change the line:

authz_cfg=Authz(auth=HTPasswdAuth(''<HTPASSWDPATH>'),

 

to

 

authz_cfg=Authz(auth=HTPasswdAuth(''/etc/krb.keytab'),

 

(Or whatever the correct path and filename is for your setup.)

 

Example:

 

###################

# STATUS TARGETS #

###################

 

<snip>

 

c['status'] = []

from buildbot.status.html import WebStatus

from buildbot.status.web.authz import Authz

from buildbot.status.web.auth import HTPasswdAuth

authz_cfg=Authz(auth=HTPasswdAuth('/etc/krb.keytab'),

        forceBuild = 'auth',

        stopBuild = 'auth',

        stopAllBuilds = 'auth',

        cancelAllPendingBuilds = 'auth',

        cancelPendingBuild = 'auth',

        gracefulShutdown = 'auth',

        forceAllBuilds = False,

        stopChange = False,

        cleanShutdown = 'auth',

        showUsersPage = 'auth',

        pauseSlave = 'auth',

        pingBuilder = False)

c['status'].append(WebStatus(http_port=8010, authz=authz_cfg))

 

You should now be prompted for your EAM/AD username and password when accessing your autobuilder.
